# AGENTS.md

## 防回归工作规范

所有参与本项目的编码代理和开发者，在修改代码或项目文件时必须遵守以下规则：

1. 修改前必须先分析问题、影响范围和可行方案，不要直接动代码。
2. 修改前必须检查 `git diff` 和 `git status`，确认当前工作区已有修改，不能覆盖他人或之前已有的修改。
3. 修复新问题时，必须主动检查相关本地文件、相似逻辑和 `REGRESSION_LOG.md`，确认不会破坏之前已经修复的问题。
4. 每次修改后必须检查 `git diff`，确认实际变更符合预期且没有无关改动。
5. 能写测试就补充回归测试，确保已修复问题不会再次出现。
6. 能运行测试就运行测试，并记录测试命令和结果。
7. 不能运行测试时，必须说明原因，并给出可执行的人工验证步骤。

## 必须执行的修复流程

每次修复问题时，必须按以下顺序执行：

1. 先阅读本文件和 `REGRESSION_LOG.md`，确认是否存在相同或相似的已修复问题。
2. 运行 `git status` 和 `git diff`，确认当前工作区状态和已有改动。
3. 使用 `rg` 或等价工具检索相关关键词、函数名、选择器、CSS 类名和错误现象，阅读相关本地文件中的现有实现。
4. 对照 `REGRESSION_LOG.md` 中的历史修复，确认本次方案不会重新引入已记录问题；如果存在相似记录，先说明关联关系。
5. 在动代码前说明问题原因、影响范围、涉及文件和修复方案。
6. 修改完成后再次检查 `git diff`，确认只包含本次预期改动。
7. 补充或更新回归测试；如果不能补测试，必须写明原因和人工验证步骤。
8. 在 `REGRESSION_LOG.md` 追加本次修复记录；如果是再次修复同类问题，必须引用之前的记录并说明为什么之前防线失效。
9. 运行 `npm run check:regression-log`，确认本地代码改动已经同步记录到 `REGRESSION_LOG.md`。
10. 交付时说明测试结果、未运行测试的原因，以及是否更新了 `REGRESSION_LOG.md`。

## 本地文件检查要求

- 解决新问题前，不能只看用户描述或当前报错点，必须检查相关本地文件和相邻实现。
- 涉及 UI、选择器、滚动、缓存、导出、目录、内容脚本等功能时，必须检索并阅读同类逻辑，避免修复一个入口却破坏另一个入口。
- 如果当前工作区已有未提交改动，必须把这些本地改动纳入影响范围分析，不能假设它们已经稳定或已经提交。
- 如果发现新问题与 `REGRESSION_LOG.md` 中已有记录相似，必须在修复说明和新记录中引用旧记录，并解释本次防回归措施。

## 使用方式

- 开始任务前，先阅读本文件和 `REGRESSION_LOG.md`。
- 修复问题后，将已修复问题、修复方式、验证方式记录到 `REGRESSION_LOG.md`。
- 提交或交付前，再次检查 `git diff` 和测试结果，确保没有引入无关改动或回归风险。
- `AGENTS.md` 和 `REGRESSION_LOG.md` 必须纳入 Git 跟踪，不能长期停留在未跟踪状态。
- 本地不 push 到 GitHub 时，也必须把每次修复写入 `REGRESSION_LOG.md`；`npm run check` 会先执行本地记录检查。
- 每次解决新问题时，必须把“已检查哪些本地文件、是否命中历史修复记录”写入修复说明或 `REGRESSION_LOG.md`。
